Documentation / HTTP API

Send events and check-ins over HTTP.

Most clients will use the Ruby, .NET, Python, CFML, or JavaScript guides, but the API itself is simple: point the client at your self-hosted Logister instance, authenticate with a project token, then POST JSON to the correct endpoint.

When to use it

Use the HTTP API when an SDK does not cover your runtime or workflow yet.

Events sent through the HTTP API show up in the same Logister views as SDK events: errors go to the inbox, logs and metrics go to activity, transaction timings power performance, and check-ins create monitor status. Use this reference when you are building a custom client, wiring a language without a first-party package, or sending one-off operational events from scripts.

Authentication

Use a project API token.

Generate the token from the target project settings page. Store it in the sending app's environment or secret manager, because Logister only shows the token once.

  • Authorization: Bearer <token>
  • X-Api-Key: <token>

Ingest events

POST application events with an event root key.

Use this endpoint for errors, logs, metrics, transactions, and check-ins when you want the full event shape.

shell
POST /api/v1/ingest_events
Authorization: Bearer <token>
Content-Type: application/json
json
{
  "event": {
    "event_type": "error",
    "level": "error",
    "message": "NoMethodError in CheckoutService",
    "fingerprint": "checkout-nomethoderror",
    "occurred_at": "2026-02-14T12:00:00Z",
    "context": {
      "environment": "production",
      "request_id": "abc123",
      "metadata": {
        "user_id": 42
      }
    }
  }
}

Event fields

Send the fields that match what you want to see in Logister.

FieldRequired?How Logister uses it
event_typeRequiredControls where the event appears. Use error, metric, transaction, log, or check_in.
messageRequiredPrimary label shown in the inbox, activity feed, and event details.
levelOptionalSeverity such as info, warn, error, or fatal. Error groups use it for severity display.
fingerprintOptionalGroups repeated errors into the same inbox issue. If absent, Logister falls back to the first message line.
occurred_atOptionalTimestamp for when the event happened. If absent, Logister uses receive time.
contextOptionalStructured metadata for investigation, filtering, release health, performance, monitor status, and related-log correlation.

For convenience, Logister also accepts common fields next to context and copies them into context when they are not already set: environment, release, trace_id, traceId, request_id, requestId, session_id, sessionId, user_id, userId, transaction_name, transactionName, duration_ms, durationMs, expected_interval_seconds, check_in_slug, monitor_slug, check_in_status, and status.

Useful context

Context is what makes the product useful after delivery succeeds.

GoalRecommended context
Make errors easier to triageenvironment, release, exception, request, user_id, and a stable fingerprint.
Link errors to nearby logstrace_id, request_id, session_id, or user_id on both error and log events.
Power transaction performancetransaction_name, duration_ms, status code, route, method, and release.
Show database loadMetric events with message set to db.query plus duration_ms, query name, and SQL context when safe to store.
Create monitor statuscheck_in_slug, check_in_status, expected_interval_seconds, environment, and optional duration_ms.

Check-ins

POST monitor heartbeats for scheduled work with a smaller payload.

Use this endpoint when the thing you are reporting is a cron job, scheduled task, or recurring worker heartbeat.

shell
POST /api/v1/check_ins
Authorization: Bearer <token>
Content-Type: application/json
json
{
  "check_in": {
    "slug": "nightly-reconcile",
    "status": "ok",
    "expected_interval_seconds": 900,
    "environment": "production",
    "release": "2026.03.02"
  }
}
FieldDefaultHow Logister uses it
slugRequiredMonitor name, such as nightly-reconcile.
statusokok keeps the monitor healthy. error marks the latest run as failed.
expected_interval_seconds300How often Logister expects the job to report before it can be considered missed.
environmentApp environmentSeparates production, staging, and other monitor streams.
duration_msBlankOptional runtime for the completed job.
release, trace_id, request_idBlankOptional context for tying a check-in back to a deploy or execution trace.

Responses

Successful writes return the Logister event id.

StatusMeaningBody
201 CreatedThe event or check-in was accepted.{"id":"event-uuid","status":"accepted"}. The ingest endpoint also includes legacy_id.
401 UnauthorizedThe API key was missing, revoked, or invalid.{"error":"Unauthorized"}
422 Unprocessable ContentThe payload shape was invalid or required fields were missing.{"errors":["..."]}

Verify delivery

Make sure your first payload shows up where you expect.

Delivery verification
1. Send one test error or transaction payload
2. Open the matching project in Logister
3. Confirm the event appears in the inbox or activity feed
4. Inspect context fields to verify the payload shape
5. Only then add the rest of your instrumentation
