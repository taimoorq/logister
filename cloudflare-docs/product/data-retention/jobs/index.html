Data retention / Jobs

Verify cleanup before trusting it on important projects.

The Data tab shows recent archive runs because users need proof that archive exports and cleanup are working. Use that visible history before relying on archive-before-delete for an important project.

Maintenance

Use manual commands when you need a controlled verification pass.

shell
DRY_RUN=true bin/rails "logister:telemetry:retention"
bin/rails "logister:telemetry:archive[ingest_events,30]"
bin/rails "logister:telemetry:archive[trace_spans,30]"
CONFIRM=prune bin/rails "logister:telemetry:prune_hot[30]"

Verify

Check the Data tab first.

  1. Use a low-risk project before changing retention on important projects.
  2. Open Project > Settings > Data and review Recent archive runs.
  3. Confirm the latest run succeeded before enabling archive-before-delete more broadly.
  4. If your team uses archive storage, confirm the matching .jsonl.gz object exists and can be opened.

Failures

Retention failures should be visible.

Failed archive runs should appear in Recent archive runs and can send retention failure notifications. If archive-before-delete is enabled, treat a failure as a stop sign: fix the archive problem first, then run cleanup again.
