Retention model

Retention is project scoped.

Changing one project's Data tab does not change another project's history. The policy controls are separated because logs and metrics, trace spans, and closed error groups often have different review value after the first few weeks.

After a policy is saved, the form collapses into a summary row. Expand it when you need to change the retention windows, turn on archive writes, or require an archive before cleanup can delete matching rows.

Archives are not a full backup

Telemetry archives are for older event history. They do not preserve user accounts, project settings, API keys, memberships, or every workflow decision. Keep ordinary backups for full restore needs.

Archive Center

Use Archive Center to check whether the plan is working.

Archive Center is the operational evidence area on the Data page. It separates the question you are asking into four paths so users do not have to scan one long settings page.

PathUse it when you need to...
OverviewConfirm whether the project is not archiving, archiving without deletion protection, protected before deletion, or blocked by a failed archive.
CoverageCheck each retention scope against the latest cleanup cutoff, archived-through date, candidate count, archived rows, and deleted rows.
CatalogReview recent archive runs, object keys, byte counts, row counts, status, and failure messages.
Search ArchivesSearch current hot telemetry first and narrow candidate archive runs for older event evidence.
