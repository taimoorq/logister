# Logister 1.1 Release Plan

Target release: `v1.1.0`
Target date: 2026-05-10
Release type: community self-hosting, distribution, and positioning release

## Release Goal

Publish the next stable Logister release with clear self-hosting distribution, permissive code licensing, protected Logister branding, and public SEO/LLM surfaces that describe the app as a forkable open source alternative for error monitoring and bug triage.

## Release Scope

- Keep the existing Rails 8 app, PostgreSQL, Redis, Sidekiq, SMTP/Amazon SES, optional ClickHouse, Fly, Cloudflare Pages, and GHCR operating model intact.
- Publish `v1.1.0` as the next GitHub Release from the top changelog entry.
- Publish `ghcr.io/taimoorq/logister:v1.1.0`, `latest`, and the short commit SHA after CI, Fly deploy, and health checks pass.
- Include the root MIT License for code use, forking, modification, self-hosting, redistribution, and commercial use.
- Include `TRADEMARKS.md` so forks and redistributed versions know they must replace Logister branding and must not imply official endorsement.
- Update README, public docs, Cloudflare docs, robots rules, `llms.txt`, and JSON-LD metadata so self-hosters, search engines, and LLM crawlers see the current release and distribution path.

## Stability Gates

Run these before merging the release commit:

```bash
git diff --check
/opt/homebrew/bin/rbenv exec bundle exec rspec spec/requests/home_spec.rb spec/system/landing_spec.rb spec/helpers/application_helper_spec.rb
```

CI should still be the full source of truth for the complete release gate, including the aggregate `CI passed` job, CodeQL/default setup on `main`, security scans, and the deploy job.

## Operational Gates

- Confirm no secrets are checked in. Runtime values must stay in Fly, GitHub Actions, Cloudflare Pages, or another deploy provider's secret store.
- Confirm the GitHub Actions workflow still has `packages: write` permission for GHCR publishing.
- Confirm `CHANGELOG.md` starts with `## v1.1.0 - 2026-05-10`; release automation reads this entry for the tag, name, and body.
- Confirm `LICENSE` and `TRADEMARKS.md` are present at the repository root.
- Confirm public docs and `llms.txt` reference `ghcr.io/taimoorq/logister:v1.1.0` as the current versioned image.
- Confirm the GHCR package remains public so self-hosters can pull release images without GitHub authentication.
- Confirm [seo-llm-measurement-runbook.md](seo-llm-measurement-runbook.md) is ready for the post-deploy discovery check.

## Release Flow

1. Commit the `v1.1.0` changelog entry, license, trademark policy, docs, SEO/LLM copy, app metadata, and tests together.
2. Push to `main` or merge the PR.
3. Wait for the aggregate `CI passed` job.
4. Let the deploy job run the Fly release command, deploy web and worker machines, and verify Fly health checks.
5. Let the deploy job publish `ghcr.io/taimoorq/logister:v1.1.0`, `ghcr.io/taimoorq/logister:latest`, and the short-SHA image tag.
6. Let the deploy job publish GitHub Release `v1.1.0` from the changelog entry.
7. Let the Cloudflare docs workflow publish static docs if `cloudflare-docs/**` changed.

## GHCR Verification

Use this after the deploy that publishes `v1.1.0`. These commands assume the GitHub CLI is authenticated as an account that can administer the `taimoorq/logister` package.

```bash
gh auth status
gh run list --workflow "CI / Deploy" --branch main --limit 1
gh release list --limit 1
gh release view v1.1.0 --json tagName,isDraft,isPrerelease,publishedAt,url
gh api /user/packages/container/logister --jq '{name, visibility, html_url}'
gh api /user/packages/container/logister/versions --jq '.[] | {id, tags: .metadata.container.tags, updated_at}'
docker pull ghcr.io/taimoorq/logister:v1.1.0
```

If the package visibility is private, open `https://github.com/users/taimoorq/packages/container/package/logister/settings` and change it to public so self-hosters can pull the image without authentication.

## Smoke Test

- Sign in and load the dashboard.
- Open a project from the top navigation dropdown.
- Open the inbox and verify assignment filters and error detail panes still work.
- Confirm archived projects stay out of active navigation while remaining reachable through archived/all project views.
- Visit the public homepage and About page and verify the open source/self-hosting positioning is visible.
- Visit the docs self-hosting Docker section and verify it references `v1.1.0`.
- Run the SEO and LLM measurement runbook after production and docs deploys complete.
