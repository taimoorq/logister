# Logister 2.0 Release Plan

Target release: `v2.0.0`
Target date: after the 2.0 beta workstream is stable
Release type: stable analytics, telemetry, and self-hosting release

## Release Goal

Ship Logister 2.0 as the stable release for the expanded observability app: error triage, logs, metrics, transactions, spans, check-ins, Insights workbooks, optional ClickHouse analytics, and optional S3-compatible cold archive storage.

## Product Direction

- Keep the Rails, Turbo, Stimulus, and server-rendered HTML architecture. Use morph refreshes, preserved scroll, stable Turbo Frame boundaries, and View Transitions for SPA-like continuity without adopting a client-side router.
- Treat PostgreSQL as the system of record, Redis as cache and job infrastructure, ClickHouse as the high-volume analytics path, and S3-compatible storage as the cold archive path.
- Make the first-party add-ons feel like one product across Ruby, .NET, Python, JavaScript, and direct HTTP/CFML integrations.
- Keep self-hosting as the primary supported deployment story while keeping `logister.org` compatible with the same configuration model.

## Highest-Impact Work Before Stable 2.0

### Insights GA

- Add saved workbook views so teams can preserve useful combinations of time range, environment, release, attributes, and selected series.
- Make Insights URLs shareable by reflecting workbook state in query params, while keeping local state as a convenience fallback.
- Add empty, loading, error, and stale-data states for every Insights panel and chart.
- Add a compact export path for the current workbook slice, starting with CSV or JSON for recent events and selected series.
- Decide which panels should read from ClickHouse first when ClickHouse is healthy, and keep PostgreSQL fallback behavior explicit.

### UI Continuity

- Keep Turbo morph refreshes enabled globally with preserved scroll.
- Use stable `view-transition-name` values on app-shell regions, project cards, inbox rows, detail panes, and Insights workbench sections.
- Use small, generic Stimulus controllers for optimistic UI state and ARIA attributes instead of styling-specific JavaScript.
- Keep motion progressive and accessible: no required animation behavior, and `prefers-reduced-motion` must remain respected.
- Add system specs for the project inbox and Insights happy path once chart rendering is stable enough for browser-level assertions.

### Storage And Cost Control

- Add an operator-facing retention view or task summary that shows hot PostgreSQL rows, ClickHouse availability, archive export status, and prune candidates.
- Add guardrails for expensive aggregate requests: bounded windows, bounded metric series counts, cache keys that include filters, and predictable defaults.
- Add archive restore notes before calling S3 archive support complete, even if restore remains a manual operator flow for 2.0.
- Track ClickHouse ingest failures separately from customer telemetry so self-monitoring cannot create noisy recursive signals.

### Add-On Parity

- Confirm Ruby, .NET, Python, and JavaScript packages all support errors, logs, metrics, transactions, spans, and check-ins with consistent naming.
- Keep language-specific auto-instrumentation opt-in where it may surprise production apps.
- Confirm all add-ons can set environment, release, service, trace ID, request ID, fingerprint, and custom attributes.
- Keep public package docs and app setup screens aligned with the exact released package versions.

### Self-Hosting And Operations

- Verify fresh Docker image boot with PostgreSQL and Redis only.
- Verify optional ClickHouse startup, schema load, health checks, and graceful fallback when ClickHouse is unavailable.
- Verify optional S3 archive settings with AWS S3 and an S3-compatible endpoint.
- Keep `.env.sample`, README, Cloudflare docs, and `llms.txt` aligned with the stable 2.0 configuration surface.
- Confirm no production secrets or live credentials are committed.

### Release Packaging

- Ensure `CHANGELOG.md` starts with the final `## v2.0.0 - YYYY-MM-DD` entry before the stable release commit.
- Publish GitHub Release `v2.0.0` as the latest stable release.
- Publish versioned images to GHCR and Docker Hub with matching `latest` tags after CI and deploy gates pass.
- Confirm beta tags remain prereleases and do not displace the stable 2.0 release.

## Smoke Test

- Sign in and load the dashboard.
- Open a project from the top navigation and verify the app shell transitions smoothly.
- Open project Insights and switch between Overview, Health, Performance, Monitors, and Custom presets.
- Filter Insights by environment, release, and an attribute; confirm the timeline, signal mix, readout, and recent stream stay in sync.
- Open the inbox, select an error row, assign it, resolve it, and verify the list and detail panes keep scroll and selection context.
- Open Activity, Performance, Monitors, and Settings for a project with real telemetry.
- Run archive export and prune tasks in a staging-like environment.
- Verify public docs, package links, and self-hosting instructions reference the stable 2.0 release.

## Go/No-Go Checklist

- [ ] Insights workbook state is stable, bounded, and shareable.
- [ ] Turbo morph refresh, View Transitions, and reduced-motion behavior pass request and system checks.
- [ ] ClickHouse is optional, observable, and does not block the core app when unavailable.
- [ ] S3-compatible archive support is documented and safe to operate.
- [ ] SDK package docs match the released add-on capabilities.
- [ ] `.env.sample`, README, Cloudflare docs, and AI-readable docs match the 2.0 runtime surface.
- [ ] Full test suite and CI release gates pass.
- [ ] GHCR and Docker Hub versioned images are public.
- [ ] GitHub Release `v2.0.0` is published as the latest stable release.
