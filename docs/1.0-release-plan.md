# Logister 1.0 Release Plan

Target release: `v1.0.0`
Target date: 2026-05-10
Release type: first stable self-hosted app release

## Release Goal

Ship Logister as a stable, self-hostable Rails 8 app for teams that want one place to ingest application errors, logs, metrics, transactions, and check-ins, then triage those signals across projects without relying on hidden hosted-only infrastructure.

This release should make self-hosters confident that the app has a coherent product surface, documented operations, tested core flows, and release automation that announces the update through GitHub Releases.

## Product Scope

- Cross-app dashboard with server-backed ECharts explorer data, needs-attention summaries, project shortcuts, and assigned-to-me links.
- Project inbox for grouped error triage with status filters, search, assignee filters, assignment controls, occurrence history, related logs, and fixed-height detail panes.
- Runtime-aware error details for Ruby, .NET, Python, JavaScript/TypeScript, and CFML events, including prominent request method and URL/path context when integrations provide it.
- Project lifecycle support for active, archived, restored, and deleted projects.
- Archived projects stay accessible from archived/all project views, disappear from active dashboards and top navigation, and revoke active API tokens.
- Project settings for API keys, team access, assignment workload counts, integration guidance, email notification preferences, and lifecycle controls.
- Project error notifications through SMTP/Amazon SES, including first-occurrence alerts and daily or weekly digests.
- Performance, activity, and monitor pages for transaction metrics, logs, custom metrics, check-ins, release health, and scheduled work.
- Static public docs on Cloudflare Pages, repo-facing docs, GHCR release images, and `llms.txt` surfaces that match the shipped product.

## Stability Gates

Run these before merging the release commit:

```bash
git diff --check
/opt/homebrew/bin/rbenv exec bundle exec rspec
/opt/homebrew/bin/rbenv exec bundle exec rubocop
/opt/homebrew/bin/rbenv exec bundle exec brakeman --no-pager --quiet --exit-on-warn --exit-on-error
```

CI must pass the aggregate `CI passed` job before deploy. The aggregate depends on Ruby scan, JavaScript scan, lint, request/model/job specs, system specs, and CodeQL default setup on `main`.

## Operational Gates

- Confirm no secrets are checked in. Runtime values must stay in Fly, GitHub Actions, Cloudflare Pages, or another deploy provider's secret store.
- Confirm the GitHub Actions workflow has `packages: write` permission so it can publish versioned release images to GitHub Container Registry.
- After the first GHCR publish, confirm the `ghcr.io/taimoorq/logister` package visibility is public if self-hosters should be able to pull images without GitHub authentication.
- Confirm `RAILS_MASTER_KEY`, `DATABASE_URL`, `DATABASE_MIGRATION_URL` when needed, `REDIS_URL`, `LOGISTER_PUBLIC_URL`, and `LOGISTER_ADMIN_EMAILS` are configured in production.
- Confirm SMTP/SES secrets are present before enabling real project notification mail: `LOGISTER_EMAIL_FROM`, `SES_SMTP_USERNAME`, `SES_SMTP_PASSWORD`, and any custom SES/SMTP variables.
- Confirm the Fly app has separate web and worker process groups. The worker must be running for Action Mailer delivery, first-occurrence alerts, digest scheduling, and optional ClickHouse writes.
- Confirm release migrations run in the Fly release phase through `./bin/rails db:prepare`, not on every web boot.
- Confirm Cloudflare Pages docs secrets and variable are present before expecting docs auto-deploy: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, and `CLOUDFLARE_PAGES_PROJECT`.

## Database And Data Safety

The 1.0 migration set includes project archive state, project notification preferences/deliveries, assignment columns on `error_groups`, and indexes for dashboard, project, inbox, assignment, and notification query paths.

Before production rollout:

- Take a database backup or verify managed PostgreSQL point-in-time recovery.
- Review migration runtime on production-like data if the production event volume is large.
- Verify indexes exist after migration for high-traffic paths:
  - `ingest_events` by project/event type/occurred time
  - `error_groups` by project/status/last seen
  - `error_groups` by project/assignee/status/last seen
  - notification delivery/preference indexes used by first-occurrence and digest jobs
- Confirm archived projects cannot ingest through old tokens and cannot generate new tokens until restored.

## Release Flow

1. Keep `CHANGELOG.md` headed by `## v1.0.0 - 2026-05-10`; the GitHub release workflow reads this entry for the tag, name, and body.
2. Commit the release plan, changelog, docs, app changes, migrations, and tests together.
3. Push to `main` or merge the PR.
4. Wait for `CI passed`.
5. Let the deploy job run `flyctl deploy --remote-only --max-concurrent 1 --wait-timeout 10m --deploy-retries 3`.
6. Let the deploy job verify Fly machine health.
7. Let the deploy job publish `ghcr.io/taimoorq/logister:v1.0.0`, `ghcr.io/taimoorq/logister:latest`, and a short-SHA image tag.
8. Let the deploy job publish GitHub Release `v1.0.0`.
9. Let the Cloudflare docs workflow publish static docs if `cloudflare-docs/**` changed.

## Post-Deploy Smoke Test

- Sign in as an admin/user.
- Load the dashboard and verify explorer charts render.
- Open a project from the top navigation dropdown.
- Create a test project, generate an API key, and send one test error.
- Confirm the error appears in the inbox and opens the detail pane.
- Assign the error to yourself, filter by `Assigned to me`, then clear the assignment.
- Archive the test project and confirm it disappears from active dashboard/top navigation while remaining accessible through archived projects.
- Restore the test project and generate a fresh API key if ingestion should resume.
- Update project email notification preferences and confirm settings save.
- Confirm `/up` responds successfully.
- Confirm `/health/clickhouse` matches the expected ClickHouse configuration state.

## Rollback Plan

- If deploy health checks fail before release publication, fix forward or redeploy the previous known-good commit through Fly.
- If the app deploy succeeds but a product regression appears, pause project notification jobs by stopping or scaling down the worker process only if email delivery is the problem; otherwise keep the worker running and deploy a fix.
- If a migration-related issue appears, restore from managed PostgreSQL recovery or backup rather than manually editing production data.
- If docs publish bad content, revert the docs commit and rerun the Cloudflare Pages workflow.
